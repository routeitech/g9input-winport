# 字庫 — `t2s.txt`

> 繁體 → 簡體 單字對照表。使用者啟用「輸出簡體」選項後，選字送出前會用這張表把
> 繁體字替換成對應的簡體字。

---

## 一、角色定位

`t2s.txt` 回答這個問題：
**「中文字 X（繁體）對應的簡體字是什麼？」**

設計上的一個重要決策：**候選字 UI 永遠顯示繁體**，只有在**送字的最後一刻**才可能做
繁→簡轉換。這樣使用者無論輸出繁體或簡體，看到的 UI 都是一致的。

### 1-1. 轉換時機

```
使用者看到候選字「華」（繁體）
   ↓
按數字鍵選字，觸發 OutputService.Send("華")
   ↓
檢查：UseSimplifiedOutput 是否為 true？
   ↓ No → 直接輸出「華」
   ↓ Yes
查 t2s.txt["華"] = "华"
   ↓
輸出「华」（簡體）
```

### 1-2. 只處理單字，不處理詞

這張表是**單字對照**，不考慮詞語上下文。例如：

- `t2s.txt["發"] = "发"`（髮也是发、頭髮的髮也是发）

對於所有繁體「發」，無論在「發展」還是「頭髮」的語境中，都轉為「发」。這會產生一個
**錯誤**：「頭髮」（繁）正確簡體應是「头发」，程式會輸出「头发」——**恰好正確**（因為
这个例子下单字對應就對）。

但：

- `t2s.txt["著"] = "着"`

這會把「著名」變成「着名」——**錯誤**，正確簡體應是「著名」（簡中「著」仍保留在書名號意思上）。

這是單字對照表的**固有限制**。若要做精確的詞庫級繁簡轉換，應使用專業的 [OpenCC](https://github.com/BYVoid/OpenCC)
而不是這張簡易表。本程式的 `t2s.txt` 定位為**輕量級的傻瓜轉換**，已知有這類邊界情況。

**沒有 `t2s.txt` 的話**：輸出簡體功能無法使用（沒有對應資料）。使用者啟用輸出簡體後
仍會輸出繁體。

DEMO 版隨附的 `t2s.txt`：

- 大小：約 20 KB
- 行數：約 2,480 行
- 涵蓋：絕大多數常用繁簡一對一字

## 二、檔案格式

### 2-1. 整體結構

純文字檔案。每行一筆：

```
<traditional> <simplified>
```

- `<traditional>` 是繁體字（一個字素）
- `<simplified>` 是對應的簡體字（一個字素）
- **分隔符是半形單一空白 `' '`（ASCII 0x20）**

例：

```
鐵 铁
繁 繁
寫 写
寬 宽
寶 宝
專 专
尋 寻
對 对
導 导
將 将
```

### 2-2. 編碼與換行

- **編碼**：UTF-8
- **換行**：`\n` 或 `\r\n`
- **空白行**：略過

### 2-3. 讀取邏輯

程式用 `line.Split(' ')` 分割：

- `parts[0]` = 繁體
- `parts[1]` = 簡體

所以每行**恰好要有一個空白**。如果你不小心寫成：

```
鐵  铁
```

（兩個空白），`Split` 會產生 `["鐵", "", "铁"]`，此時 `parts[1] = ""`（空字串），
這筆資料被忽略（程式會跳過空映射）。

### 2-4. 繁體和簡體「相同」的字

有些字在繁簡中寫法**完全一樣**（如「中」「國」「人」「山」「水」）。這些字**不必**
寫入 `t2s.txt`——不寫的效果等同於寫 `中 中`。寫了只是浪費行數。

但 DEMO 版的表裡確實有一些 `繁 繁` 形式的條目（如 `繁 繁`）——這是為了避免某些字
被誤會成有對應簡體版本而寫下的「穩定化」條目。你可以保留，也可以刪掉都不影響行為。

### 2-5. 重複 key

後到覆蓋（`Dictionary` indexer）。

### 2-6. 一繁多簡、一簡多繁

中文的繁簡對應**不是一對一**。例如：

- 繁「後」 → 簡「后」（時間後方）
- 繁「皇后」 → 簡「皇后」（「后」本來就兩邊都有）

這種一簡對多繁的情況，在**繁→簡**方向上不成問題（只需把「後」映射到「后」）。
但在**簡→繁**方向上就很麻煩——「后」該映射回「後」還是「皇后」？需要上下文。

本程式**只做繁→簡**，不做簡→繁，所以避開了這個複雜問題。

## 三、key 設計原則

### 3-1. 涵蓋範圍

**至少包含**：`char_map.txt` 裡所有會出現的繁體字。

**更好的做法**：包含所有**常用繁體字**（約 6,000~8,000 字），這樣萬一使用者透過其他
途徑（如右鍵同音字、關聯字）輸出了非 `char_map.txt` 裡的字，也能正確轉換。

### 3-2. 不處理 HKSCS 方言字

HKSCS 方言字（「嘅」「喎」「冇」）**在簡體中文裡沒有對應**——簡體中文不寫這些粵語字。
做法有兩種：

- **不寫入 `t2s.txt`**：使用者啟用輸出簡體時，這些字原樣輸出為繁體（方言字）
- **寫入映射到「自身」**：`嘅 嘅`，效果同上但更明確

兩種都可，建議後者（明確性較高）。

### 3-3. 不處理標點

全形標點在繁簡中共用（「，」「。」「：」等），不需要轉換。如果你的字庫裡有繁體特有
標點（如「『』」繁對應「""」簡），可考慮加入，但這並非必要。

## 四、value 設計原則

### 4-1. 採用中國大陸官方簡化字

推薦遵循中華人民共和國國家語言文字工作委員會 1986 年發布的《簡化字總表》。
這是最廣泛被接受的簡化字標準。

### 4-2. 不要自創簡化

不要自行把「繁」改成「烦」等**不是官方的簡化**——這會讓你的輸出在中國大陸看起來
詭異。

### 4-3. 特殊字的處理

有些字在官方規範中有**多個合法簡化**（例如「乾」可簡為「干」或「乾」，視語境而定）。
本表只能選一個，建議選**最常用**的：

- 「乾」 → 「干」（對應「乾淨」「乾脆」等大多數用法）
- 「乾」如果原本用於「乾卦」「乾隆」等，簡體仍保留「乾」

沒有完美的單字表能處理這種情況。如果你的使用者對繁簡轉換品質要求高，**建議不要
啟用輸出簡體功能**，改為讓使用者手動用 OpenCC 或類似工具做後處理。

## 五、載入行為與實作細節

### 5-1. 載入時機

啟動時載入，`FrozenDictionary<string, string>`。

### 5-2. 查詢時機

`OutputService.Send(char)` 中，若 `UseSimplifiedOutput == true`：

```csharp
public string ConvertForOutput(string text)
{
    if (!UseSimplifiedOutput) return text;

    var sb = new StringBuilder();
    foreach (var grapheme in ParseGraphemes(text))
    {
        if (T2sMap.TryGetValue(grapheme, out var simplified))
            sb.Append(simplified);
        else
            sb.Append(grapheme);
    }
    return sb.ToString();
}
```

按 grapheme 逐字轉換。對於沒在表中的字，保持原樣。

### 5-3. 與 PUA 轉換的順序

**重要**：`b5_uni.txt` 的 PUA 轉換發生在**繁簡轉換之前**。流程是：

```
選字：候選 = "嘅"（可能 PUA）
  ↓
Step 1: PUA → Unicode（用 b5_uni.txt）
  結果：仍是「嘅」U+55CE（標準 Unicode）
  ↓
Step 2: 繁→簡（用 t2s.txt）
  查 t2s.txt["嘅"] → 不存在（方言字不該有簡體對應）
  結果：保持「嘅」
  ↓
送字
```

### 5-4. 使用者介面：切換簡繁

使用者可以在：
- 托盤選單：右鍵圖示 → 「輸出簡體」切換
- 導覽列：最左邊的「繁／簡」按鈕

切換後立刻生效（不需重啟）。UI 上候選字仍是繁體，**只是輸出時轉換**。

## 六、與其他字庫的關係

### 6-1. 與 `char_map.txt`

`char_map.txt` 的候選字清單應都是**繁體**。如果你把簡體字放入 `char_map.txt`，會導致：

- 候選字 UI 顯示簡體（對繁體使用者不友善）
- `t2s.txt` 查詢會失敗（因為簡體字不是繁體 key）
- 切到輸出簡體時變成簡→簡的無效查詢，保持原樣

### 6-2. 與 `rel_map.txt`

關聯字表也應該用繁體作為 key 和 value。同理。

### 6-3. 與 `cj5_map.txt` / `b5_map.txt`

轉倉／轉易模式下，**繁簡轉換不會發生**——因為送出的是字碼（VK 事件）給系統 IME，
系統 IME 會依自身設定組字（如果使用者用的是繁體倉頡，組出來就是繁體）。

換言之：**繁簡轉換只在「剪貼簿」和「直出」兩種輸出模式下有效**。

如果使用者同時啟用「輸出簡體」和「轉倉」，程式的實際行為是：**繁簡轉換被跳過，
送出繁體字碼**。這是合理的，因為轉倉本來就是在模擬繁體倉頡的輸入情境。

## 八、作者的實務建議

### 8-1. 直接用 OpenCC 的資料

**最推薦**：從 [OpenCC](https://github.com/BYVoid/OpenCC) 的 `TSCharacters.txt`
（繁→簡單字對照）抽取資料：

```python
# OpenCC 格式範例：
# 繁 簡1 簡2 ...
# 若有多個簡體候選，第一個是最常用

with open("TSCharacters.txt", encoding="utf-8") as f_in, \
     open("t2s.txt", "w", encoding="utf-8") as f_out:
    for line in f_in:
        line = line.strip()
        if not line or line.startswith("#"):
            continue
        parts = line.split("\t")
        if len(parts) >= 2:
            traditional = parts[0]
            simplified = parts[1].split(" ")[0]  # 只取第一個候選
            f_out.write(f"{traditional} {simplified}\n")
```

### 8-2. 字集對齊

```python
# 確保 char_map.txt 的所有字都有 t2s.txt 對應
chars_in_char_map = set(...)

t2s_keys = set()
with open("t2s.txt", encoding="utf-8") as f:
    for line in f:
        parts = line.strip().split(" ", 1)
        if len(parts) == 2:
            t2s_keys.add(parts[0])

missing = chars_in_char_map - t2s_keys
print(f"{len(missing)} chars missing in t2s.txt:")
print("".join(sorted(missing)))
```

### 8-3. 要不要做詞級轉換？

**本程式不支援**，也不建議你用這個機制做詞級轉換。原因：

- `t2s.txt` 的格式是單字對單字，key 是 1 個 grapheme
- 即使你把 key 改成多字（如「著作」），程式讀取後仍按 grapheme 切分，查不到多字 key

若要詞級轉換，使用者應該：

1. 用九格輸入法打出繁體輸出
2. 將繁體文本複製到其他工具（如 OpenCC 的線上服務、LibreOffice 擴充功能）做
    詞庫級繁簡轉換

## 九、常見錯誤診斷

| 症狀 | 可能原因 |
|---|---|
| 切到輸出簡體後仍輸出繁體 | `t2s.txt` 裡沒有該字的對應，或表根本沒載入 |
| 某個字繁簡轉換錯誤（如「著」變「着」） | 單字表的固有限制，某字有多個簡體候選但表中選了錯的 |
| 轉倉／轉易模式下簡體設定沒生效 | 正常——這兩模式下不做繁簡轉換 |
| 檔案修改後沒生效 | 沒重啟程式 |
| 簡體字的 HKSCS 方言字顯示方塊 | 方言字在簡體系統中沒對應，保留原繁體方言字，需使用者安裝 HKSCS 字型 |

## 附錄：完整字庫設計回顧

到這裡你已經看完了全部 7 份字庫的設計文件。七張表的角色總結：

| 表 | 功能 | 核心性 | 格式特色 |
|---|---|---|---|
| `char_map.txt` | 九宮格碼→候選字 | 核心必需 | 逗號分隔，支援 `*` 佔位 |
| `rel_map.txt` | 字→下一字關聯 | 常用 | 無分隔符，不支援 `*` |
| `cj5_map.txt` | 字→倉頡碼 | 選用（轉倉） | 無分隔符，code 為 a~z |
| `b5_map.txt` | 字→大易碼 | 選用（轉易） | 無分隔符，code 可含符號 |
| `homo_map.txt` | 字→粵語同音字 | 選用 | 空白分隔 |
| `b5_uni.txt` | Big5↔Unicode | 選用（HKSCS） | 逗號分隔，4 位 Big5 + 4/5 位 Unicode |
| `t2s.txt` | 繁→簡 | 選用 | 空白分隔 |

### 三種分隔符的選擇邏輯

- **逗號 `,`**：當 key 是數字序列時（`char_map.txt`），或兩邊都是固定長度十六進位
    時（`b5_uni.txt`）。
- **空白 ` `**：當 key 和 value 都是單一 CJK 字元，但 key 可能是數字時
    （`homo_map.txt`），或兩邊都是單一字元時（`t2s.txt`）。
- **無分隔**：當程式用「取第一個字素」作為 key，剩下是 value 時（`rel_map.txt`、
    `cj5_map.txt`、`b5_map.txt`）。這種作法可讓 value 任意長且不需擔心 value 裡含有
    分隔符。

### 三種重複 key 處理的選擇邏輯

- **後到覆蓋**（`char_map.txt`、`cj5_map.txt`、`b5_map.txt`、`b5_uni.txt`、`t2s.txt`）：
    允許使用者以「疊加」方式覆寫表，底下可放新條目覆蓋前面的。
- **首次生效**（`rel_map.txt`、`homo_map.txt`）：資料較大、覆寫少見；一般使用者只會
    從頭讀表，後面的重複是資料錯誤而不是意圖覆寫。

這些差異是程式內部 `Dictionary.TryAdd` vs indexer 賦值的選擇，設計上有其考量，了解後
維護起來會更順手。
