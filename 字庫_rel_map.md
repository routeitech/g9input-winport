# 字庫 — `rel_map.txt`

> 關聯字表。當使用者送出一個字後，這張表決定下一個「最有可能接在後面」的 9 個字
> 要顯示在哪裡。

---

## 一、角色定位

`rel_map.txt` 回答這個問題：**「使用者剛送出了字 X，下一個字最有可能是什麼？」**

這個功能在使用者介面上有兩種呈現：

1. **標準尺寸（AssociatePreview 狀態）**：9 格的左上角顯示小字預覽，使用者點小字即可
    直接輸出，不必打字碼
2. **迷你尺寸**：不顯示預覽（空間不夠），但仍可按 `0` 進入「關聯字選字」模式，把
    9 個關聯字放大到 9 格裡挑

**沒有 `rel_map.txt` 的話**：程式仍可運作，只是沒有關聯字提示，使用者必須一字一字打碼。

---

## 二、檔案格式

### 2-1. 整體結構

純文字檔案。**每行**一個字及其關聯字串：

```
<head><nextChars>
```

- `<head>` 是**一個字素**（grapheme），通常是單一 CJK 字元
- `<nextChars>` 是**一個或多個字元**，表示「`<head>` 之後常接的字」，順序 = 格位

注意：`<head>` 和 `<nextChars>` 之間**沒有分隔符**！程式以「第一個字素」作為 key，
剩下的全部作為 value。

例：

```
我你這好到資就個能的
```

解析：

- `<head>` = `"我"`（第一個字素）
- `<nextChars>` = `"你這好到資就個能的"`（剩下全部）

意義：「送出『我』後，下一個字最可能是『你』（第 1 格），其次是『這』（第 2 格）…」

### 2-2. 編碼與換行

- **編碼**：UTF-8（可帶 BOM）
- **換行**：`\n` 或 `\r\n` 皆可
- **空白行**：略過
- **行首 / 行尾空白**：**避免**（會被當成 head 或 chars 的一部分）

### 2-3. nextChars 的格位對應

和 `char_map.txt` 的規則完全相同：

| `<nextChars>` 中的位置 | 顯示格位 | 按鍵 |
|---|---|---|
| 第 1 個字素 | 左下 | numpad 1 |
| 第 2 | 中下 | 2 |
| 第 3 | 右下 | 3 |
| 第 4 | 左中 | 4 |
| 第 5 | 正中 | 5 |
| 第 6 | 右中 | 6 |
| 第 7 | 左上 | 7 |
| 第 8 | 中上 | 8 |
| 第 9 | 右上 | 9 |

超過 9 字的部分會進入第 2 頁、第 3 頁…… 但**注意**：在 AssociatePreview 狀態下的
**小字預覽**只顯示前 9 個（不翻頁）。如果使用者按 `0` 進入「關聯字選字」模式，
才可翻頁看全部。

### 2-4. `*` 佔位符

**不支援 `*` 佔位**。如果你想跳過某個格位，只能把該格位設成一個實際字元。

這是和 `char_map.txt` 最大的差異。設計原因：關聯字本質上是「按頻率排」，頻率排序後
自然填滿前幾格，不該出現中間空洞；若某字沒有 9 個常見關聯字，就少寫幾個即可，不需佔位。

### 2-5. 重複 head 怎麼處理

```
我你這好到資就個能的
我想要是在
```

第一筆會保留，第二筆會被忽略。程式讀取邏輯是 `Dictionary.TryAdd`——**先到先贏**。

**請注意**：這和 `char_map.txt` 的「後到覆蓋」相反！如果你想更新某字的關聯字，
必須修改原本那一行，不能在底下多加一行。

### 2-6. head 可以是多字元嗎？

理論上可以——`<head>` 是用 `StringInfo.GetNextTextElement` 取出的第一個 grapheme。
如果你寫：

```
𠀀你好
```

`<head>` 會被解析為 `"𠀀"`（U+20000，CJK 擴展 B 的 1 個字素，佔 2 個 UTF-16 char），
`<nextChars>` = `"你好"`。這對支援冷僻字很重要。

**不能**把 head 寫成多個字元（例如雙字詞）。程式只取第一個字素。

## 三、key 設計原則

### 3-1. 涵蓋範圍：哪些字應該有關聯

建議優先包含的字：

1. **高頻起頭字**：這、那、我、你、他、的、了、是、在、有…
2. **常見詞語的前字**：想（想想、想到）、去（去做、去玩）、今（今天、今年）
3. **方向／數量詞前字**：一、二、三、千、萬…
4. **情緒詞前字**：好、很、真、太…

低頻字不必收入。DEMO 約 3,600 字的規模已涵蓋絕大多數日常用字的關聯需求。

### 3-2. head 必須是 `char_map.txt` 裡的字

嚴格來說沒有這個要求——程式只要使用者**送出了**字 X，就會查 `rel_map.txt["X"]`。
但實務上使用者只會送出 `char_map.txt` 裡的字，所以：

- 在 `char_map.txt` 外的字寫進 `rel_map.txt` 是**浪費記憶體**
- 在 `char_map.txt` 內的字**不在** `rel_map.txt` 裡 → 送出後直接回 Home（沒有關聯預覽）

## 四、value 設計原則

### 4-1. 頻度優先（再次強調）

`<nextChars>` 的前 9 個字顯示在主畫面，是最該精心挑選的位置。按你的目標語料的
「X 後接 Y」共現頻度排序。

### 4-2. 用語料庫的實作

最簡單的語料庫做法：

```python
# 簡化版：讀一份語料，統計 char → next_char 的頻度
from collections import Counter, defaultdict

with open("corpus.txt", encoding="utf-8") as f:
    text = f.read()

bigram = defaultdict(Counter)
for i in range(len(text) - 1):
    a, b = text[i], text[i+1]
    # 可過濾掉標點符號
    if "\u4e00" <= a <= "\u9fff" and "\u4e00" <= b <= "\u9fff":
        bigram[a][b] += 1

with open("rel_map.txt", "w", encoding="utf-8") as out:
    for head, counter in sorted(bigram.items()):
        tops = [c for c, _ in counter.most_common(30)]  # 取前 30 字
        if tops:
            out.write(f"{head}{''.join(tops)}\n")
```

建議收錄**前 30 字**而非只有 9 字——因為多餘的部分可以進入第 2 頁、第 3 頁，讓進階
使用者能翻頁找更多候選。

### 4-3. 特殊處理：標點之後

許多使用者會在標點後立刻開始打字，所以「。」「，」「？」「！」這類標點作為 head，
應該列出**常見句首字**：

```
。（這、那、我、你、他、在、有、不、為）
```

注意：你要確保 `char_map.txt` 裡這些標點都存在（在 `0,...` 或 `09,...` 行）。

### 4-4. 涵蓋 HKSCS 和擴展字元

`<head>` 和 `<nextChars>` 都允許任何 Unicode 字元，所以你可以在關聯字裡放：

- HKSCS PUA 字元（顯示為紅色，輸出時自動轉換）
- CJK 擴展 B~F 的冷僻字
- 全形符號

## 五、載入行為與實作細節

### 5-1. 載入時機

與 `char_map.txt` 相同：程式啟動時一次性載入，凍結為 `FrozenDictionary<string, string>`
供查詢。修改檔案需重啟程式。

### 5-2. 查詢時機

每當使用者選字送出（`HandleNextWord` 成功執行），程式會：

```csharp
if (RelatedChar.TryGetValue(sentChar, out var nextChars))
{
    // 把 nextChars 的前 9 格做成關聯字列表
    for (int i = 0; i < 9; i++)
        Cells[i].CornerText = i < nextChars.Length ? nextChars[i] : "";
    // 翻頁需要的完整列表存在 _associateWords
    _associateWords = ParseGraphemes(nextChars);
}
else
{
    // 沒有關聯字 → 直接回 Home
    EnterHome();
}
```

實際上 `_associateWords` 的列表會被整個程式持續保留，即使使用者之後切到別的狀態
（例如 Home → FirstDigit），只要還沒送出下一字，原本的關聯預覽就一直在。

### 5-3. 字素解析

和 `char_map.txt` 一樣用 `StringInfo.GetNextTextElement`，正確處理 surrogate pair。
你可以放心在關聯字裡塞「𠁍」這種擴展字元。

## 六、與其他字庫的關係

### 6-1. 與 `char_map.txt`

**理想情況**：`rel_map.txt` 的每個 `<head>` 都出現在 `char_map.txt` 的某個 code 下。
否則使用者永遠不會「送出」這個字，寫了也沒用。

**實務**：沒有嚴格的一致性檢查工具，你需要自己確認。一個簡易的 Python 腳本：

```python
# 找出 rel_map 裡但不在 char_map 裡的 head
char_map_chars = set()
with open("char_map.txt", encoding="utf-8") as f:
    for line in f:
        if "," in line:
            char_map_chars.update(line.split(",", 1)[1].rstrip("\n"))

orphan = set()
with open("rel_map.txt", encoding="utf-8") as f:
    for line in f:
        if line.strip():
            head = line[0]  # 簡化版，真實情況要用 grapheme 處理
            if head not in char_map_chars:
                orphan.add(head)

print(f"{len(orphan)} orphan heads in rel_map.txt:")
print("".join(sorted(orphan)))
```

### 6-2. 與 `t2s.txt`

**特別注意**：候選字 UI 永遠顯示繁體，但送出時可能被轉為簡體。這會造成一個邊界情況：

- 使用者送出「華」（繁體）
- `t2s.txt["華"] = "华"` → 實際輸出「华」
- **但** `rel_map.txt` 的 key 是用送出前的繁體「華」來查

這是故意的設計。程式的查詢以 UI 層的繁體為準，轉簡只在最後輸出那一刻發生。
所以你不必另外維護一份簡體的 `rel_map.txt`。

### 6-3. 與 `homo_map.txt`

`homo_map.txt` 也有**由字查關聯**的特性（右鍵查同音字）。但兩者作用完全不同：

- `rel_map.txt`：語義上「下一個字是什麼」（bigram 頻度）
- `homo_map.txt`：語音上「同音字」（粵語讀音分組）

不會互相干擾。

## 七、實例分析

DEMO 版 `rel_map.txt` 的開頭：

```
我你這好到資就個能的
一個樣次些種齊般定起
丁丑香美興文
七律十月年公七倍大零
乃至是
九十五三九月方百倍州
了不解得了事斷然結五
二線流戰日號期萬百回
人家員民生們類物士才
入境學門侵選法會場睡
八百十九月年字倍戒方
刀子劍鋒槍法鞘直切闊
力氣量不學度圖爭行竭
匕首戲
十分份五年二三月足倍
又是有要在來如會叫吃
```

逐行分析：

| head | 關聯字（前 9）| 解讀 |
|---|---|---|
| 我 | 你、這、好、到、資、就、個、能、的 | 「我你…」「我這…」「我好…」「我到…」等常見組合 |
| 一 | 個、樣、次、些、種、齊、般、定、起 | 「一個」「一樣」「一次」… |
| 丁 | 丑、香、美、興、文 | 只收了 5 字，沒填滿 9 格 |
| 七 | 律、十、月、年、公、七、倍、大、零 | 注意「七」本身作為關聯字的第 6 個（「七七事變」之類的重複） |
| 乃 | 至、是 | 只 2 字 |
| 了 | 不、解、得、了、事、斷、然、結、五 | 「了了」（去去了了）、「了解」、「了斷」… |
| 又 | 是、有、要、在、來、如、會、叫、吃 | 「又是」、「又有」… |

觀察：

- 多數高頻字有完整 9 字
- 罕用字（如「丁」「乃」「匕」）只有 2~5 個關聯字，這是合理的
- 關聯字中可以包含 head 本身（如「七」的關聯字第 6 位是「七」，用於「七七事變」這類疊字）

## 八、作者的實務建議

### 8-1. 先做核心 500 字，再擴展

最常用的 500 個起頭字覆蓋了 80% 以上的輸入情境。依下列順序擴展：

1. 核心 500（的、一、是、我、你、他、這、了、在…）
2. 日常 2000（加上情緒、動作、名詞）
3. 進階 5000（加上專業詞彙、地名、人名）
4. 完整字庫（所有可能出現的字）

### 8-2. 定期用真實文本驗證

找一份你常寫的文本（email、日記、技術文件），用語料庫的方式統計 bigram，
把結果和你的 `rel_map.txt` 比對，找出排序不合理的項目改善。

### 8-3. 不要過度追求涵蓋

每個 head 其實有 30~50 個次字（第 2、3 頁）就很夠用了。超過 100 字的關聯字列表反而
讓使用者翻頁翻到崩潰。

### 8-4. 使用內建工具「查看關聯字」功能

目前程式沒有內建查看關聯字的功能（這是一個未來可以加入的工具）。DIY 方法：
暫時把使用者很常送的字作為 head，送出一次看 UI 顯示的關聯字是否合理，不合理就改 `rel_map.txt`。

## 九、常見錯誤診斷

| 症狀 | 可能原因 |
|---|---|
| 送字後沒有關聯預覽出現 | 該字不在 `rel_map.txt` 裡，或關聯字為空 |
| 關聯預覽的字順序和預期不同 | 檢查 `<nextChars>` 裡的字元順序 |
| 關聯預覽只顯示某幾個格子，其他空白 | `<nextChars>` 本身就不足 9 字，正常 |
| 關聯字跳到錯誤的格位 | 某些看起來像 1 字的其實是 2 字（如組合字符），用 grapheme 解析檢查 |
| 改了檔案但關聯字不變 | 沒有重啟程式，或該 head 已有舊行、新行沒有覆蓋 |
| 某字 head 有兩行都寫了 | 只有第一行生效（與 `char_map.txt` 規則相反） |

## 十、快速參考卡

```
格式：<head><nextChars>（無分隔符）
編碼：UTF-8
head：一個字素（支援 surrogate pair）
nextChars：任意字元串，順序 = 格位 1~9，超過 9 字進入後續頁面
不支援：* 佔位符
重複 key：首次生效（與 char_map.txt 相反）
```
