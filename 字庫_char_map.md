# 字庫 — `char_map.txt`

> 九宮格字碼對照表。這是整個輸入法的**核心資料表**，決定使用者打什麼碼會看到什麼候選字。
> 是 7 份字庫裡最重要、也最複雜的一份。

---

## 一、角色定位

`char_map.txt` 回答這個問題：**「當使用者輸入碼 X（1~3 位數字）時，要在九宮格顯示哪 9 個字？」**

如果這張表不存在或是空的，輸入法變成完全沒用的九宮格畫面——使用者按任何鍵都不會有候選字
跳出來。其他 6 份字庫都是錦上添花，只有這份是不可或缺。

---

## 二、檔案格式

### 2-1. 整體結構

純文字檔案。**每行**是一筆資料，格式為：

```
<code>,<chars>
```

- `<code>` 是 1~3 位的 ASCII 數字（`0`~`9`），代表使用者輸入的九宮格碼
- 逗號 `,` 是**半形英文逗號**（ASCII 0x2C），**不是**全形逗號「，」
- `<chars>` 是一個沒有分隔符的 CJK 字元串，每個字元就是一個候選字

例：

```
110,華萬夢若著花落英苦幕蘭舊草莫...
```

這行的意思：「當使用者輸入 `110` 時，候選字依序是『華、萬、夢、若、著、花、落、英、苦、
幕、蘭、舊、草、莫…』」——注意字在 `<chars>` 中的**順序**等同於選字時的顯示順序。

### 2-2. 編碼與換行

- **編碼**：UTF-8。可帶 BOM（EF BB BF），程式自動偵測並跳過
- **換行**：支援 `\n`（Unix）或 `\r\n`（Windows）都可
- **空白行**：允許存在，會被載入程式直接略過
- **行首空白**：**不要**在行首加空白，會使得程式把 space 當成 code 的一部分
- **行尾空白**：不要在行尾加多餘空白或 tab，會被當成 chars 的一部分

### 2-3. key 的四種類別

`<code>` 只有 4 種合法類型，每種有特殊語意：

| 類別 | 位數 | 範例 | 語意 |
|---|---|---|---|
| **標點** | 1 | `0` | 標點模式下第一頁顯示的 9 個常用標點 |
| **擴充標點** | 2 | `09` | 標點模式按「下一頁」後進入的擴充標點集 |
| **百家姓** | 2 | `10`, `20`, ..., `90` | 首碼為 1~9 時按「0」進入的姓氏清單 |
| **字** | 3 | `110`, `111`, ..., `999` | 一般字的 3 碼拆碼 |

**`0` 以外的 1 碼 key（如 `1`, `2`, ...）、`00`~`09` 以外的 2 碼 key（如 `11`, `12`）通常
不會出現在這張表裡**——它們是使用者拆碼過程中的中間狀態，程式會自動查 `Prefix2Cache`
（由 3 碼 key 聚合出來），不需要你手動寫。

> 例外：理論上你可以寫 `12,某某某` 這樣的 2 碼 key，但程式的目前行為是：只有 `X0` 這種
> 「第二碼為 0」的才會被當成百家姓觸發。一般 2 碼 key 不會在 UI 上以特殊方式呈現，
> 寫了也只是浪費記憶體。建議不要寫。

### 2-4. chars 中的特殊字元：`*` 佔位符

在 `<chars>` 中可以出現星號 `*`，表示**「這個位置的候選字保留為空」**。舉例：

```
123,**期*
```

這行表示：「碼 `123` 的候選字：第 1 格空、第 2 格空、第 3 格是『期』、第 4 格空、第 5~9 格不存在」。

為什麼要有 `*`？因為**候選字的順序等於格子位置**：

- 第 1 個字元 → 顯示在 numpad `1` 的格子（左下）
- 第 2 個字元 → numpad `2`（中下）
- 第 3 個字元 → numpad `3`（右下）
- 第 4 個字元 → numpad `4`（左中）
- 第 5 個字元 → numpad `5`（正中）
- 第 6 個字元 → numpad `6`（右中）
- 第 7 個字元 → numpad `7`（左上）
- 第 8 個字元 → numpad `8`（中上）
- 第 9 個字元 → numpad `9`（右上）
- 第 10 個字元及之後 → 第二頁（也按格位對應）、第三頁…

如果你想讓**字「期」固定出現在第 3 格位置**，但前兩格沒有適合的字，就必須用 `**` 佔位。
程式內部會把 `*` 在顯示時轉成灰色空格，不會被誤選。

### 2-5. 超過 9 字怎麼辦？

`<chars>` 的長度**沒有硬上限**。超過 9 字的部分會自動分頁，使用者可按 `+` / `-` 翻頁。

例：

```
0,！：；。，、—？…
```

這個 1 碼的「標點」只有 9 字一頁，剛好。

```
09,（擴充標點約 371 個字元）
```

2 碼的「擴充標點」有 371 字，程式算出有 371/9 ≈ 42 頁，使用者可用 `+` 翻到下一頁。

### 2-6. 一個 code 多行 → 只讀最後一行

如果同一個 code 重複出現：

```
110,甲乙丙
110,丁戊己
```

程式會**只保留最後一行**（`110` = 「丁戊己」）。這是 `Dictionary` 的標準覆寫行為。
這個性質可用於：先載入一份「基礎表」，後面再加一份「補充表」覆蓋其中部分項目。

### 2-7. 第 1 碼從哪個字元開始？

`code` 字串的第一個字元 = 使用者按下的第 1 碼。這有一個特殊情況：**`0` 不能作為第 1 碼**。
如果你寫：

```
012,某字
```

這個 code 永遠不會被觸發，因為 `0` 在 Home 狀態的作用是「進入標點」而不是「開始一個
字碼」。

合法的第 1 碼只有 `1`~`9`。`0` 只能在 code 的**第 2 碼位置**出現（觸發百家姓）或作為
1 碼 code 本身（`0` 觸發標點，`09` 觸發擴充標點）。

## 三、key 設計原則（拆碼思路）

### 3-1. 9 種基本筆型的分派

預設字庫遵循「九方」流派的筆型分派：

| 碼 | 筆型 | 代表視覺 |
|---|---|---|
| 1 | 交叉（十字型） | 十、古、早、占、畢 |
| 2 | 交叉（斜乂型） | 乂、父、交、文 |
| 3 | 勾／帽（上包覆） | 冂、內、同、冠、周 |
| 4 | 點 | 心、冰、火、氵、丶 |
| 5 | 折／乙 | 乙、已、己、鳥、馬 |
| 6 | 方／口 | 口、國、回、田、因 |
| 7 | 橫 | 一、十、上、七、平 |
| 8 | 豎 | 中、十、下、丨、申 |
| 9 | 撇 | 人、川、斜、斤、乃 |

你可以**自訂**不同的分派——例如把折和勾合併、把點和橫合併，空出鍵位給其他分類。
這完全由你的 `char_map.txt` 決定，程式不在乎數字到筆型的語意對應。

### 3-2. 第 1 碼取字原則

看整個字，找**最左上方**的那個筆劃／部件：

| 字 | 最左上 | 第 1 碼 |
|---|---|---|
| 華 | 艸 (橫) | 1（十）或 7（橫），實務上選 1 更合理因為艸 = 屮+屮 近十字型 |
| 我 | 丿 (撇) | 9（撇） |
| 你 | 亻 (單點或斜) | 9（撇）或 2（斜乂） |
| 好 | 女 (左邊部首) | 9（撇）或 2 |
| 國 | 囗 (方框) | 6（方） |
| 高 | 亠 (橫上一點) | 4（點）或 7 |

視覺判斷的**優先權**（上從高到低）：

1. 如果字是「包覆型」（囗、匚、冂等），先看外框 → 選對應筆型
2. 如果字是「左右型」，看左邊部件的起筆
3. 如果字是「上下型」，看上半部件的起筆
4. 獨體字直接看整字最左上的筆劃

### 3-3. 第 2 碼取字原則

在已取第 1 碼的前提下，看**剩下部件**的**最右上方**筆劃／部件。

| 字 | 第 1 碼 | 剩下部分 | 第 2 碼 |
|---|---|---|---|
| 華 | 1 | 「華-艸」＝「十十一十」 | 1（十） |
| 林 | 9 | 「林-朩」＝「朩」 | 9 |
| 好 | 9 | 「好-女」＝「子」 | 1（子的起筆是十字型）|

第 2 碼**不是**「字的次筆」，而是「取完第 1 碼後剩下部件最顯眼的筆劃」。

### 3-4. 第 3 碼取字原則

第 3 碼是**字的最後一筆**或**最右下方的收尾筆劃**。這一碼的作用是「分辨同首 2 碼的不同字」。

例：首碼 1 + 次碼 1 下有許多字，第 3 碼幫忙分類：

- `110` = 最末筆不明顯／通用（華、萬、夢、若、著、花、落…）
- `111` = 最末筆為十字型（著、蓋、萊、藝、苦…）
- `112` = 最末筆為斜乂（艾、勸、惹、若、英…）
- `113` = 最末筆為包覆／勾（繭、藤、藍、蘭、蕭…）
- `117` = 最末筆為橫（苛、蕾、菲、華、葬…）
- `118` = 最末筆為豎（芷、蘋、蔽、蕭、菲…）
- `119` = 最末筆為撇（擎、敬、警、驚、鸛、歡…）

實務：`110` 是個 **「全收夾」**——所有首碼 1 + 次碼 1 的字都先收進來，然後再從 111~119
各自挑該組別的字再放一次，便於使用者用 `110` 粗查、用 `11X` 精查。

### 3-5. 常見碼的命名慣例

- `1N0` ~ `9N0`（末碼為 0，即 `N0`+另一碼）：**百家姓**（`10`, `20`, ..., `90`）
- `N10` ~ `N99`（未出現 0）：一般字碼
- `1N9` 的字通常是末筆為撇的字

## 四、value 設計原則（排列順序）

### 4-1. 順序就是視覺位置

重申：`<chars>` 裡的**順序 = 候選字格位**。第 N 個字元（以 `StringInfo.GetNextTextElement`
計算 grapheme，即支援輔助字面的 4-byte 字元）出現在第 N 格。

### 4-2. 頻度優先原則

建議字庫作者把**使用頻率最高**的字放在前 9 個位置（第一頁）。剩下的放在第 10 個後
（第二頁之後，需要使用者翻頁）。

例：

```
110,華萬夢若著花落英苦  幕蘭舊草莫藝葉薦藍藏蓋藥蔡...
    |─────第 1 頁──────|  |─────第 2 頁以後──────|
```

### 4-3. 字頻排序方法

如果你有語料庫，可以按：

1. 對目標語料做字頻統計（例如中研院的中文詞語料庫）
2. 排序候選字
3. 前 9 個放第一頁，其餘放後頁

沒有語料庫的話，參考 DEMO 版內建的順序是一個合理起點。

### 4-4. `*` 佔位符的正確使用

如果你發現某個碼底下，某個**格位**沒有合適的字（尤其是 1~3 格，也就是下排），
可以用 `*` 佔位，避免把其他字推到錯誤的位置。

```
123,**期*
```

這種寫法保證：

- 格 1 空
- 格 2 空
- 格 3 顯示「期」
- 格 4 空
- 格 5~9 完全不存在（因為超出字串長度）

但這也帶來一個副作用：當使用者打 `123` 時 UI 會大部分是空的，觀感不佳。
實務上如果你有更多字，應該把它們補上：

```
123,甲乙期丙丁戊己庚辛
```

### 4-5. 候選字裡不應出現的字元

- **空白字元（space, tab, 全形空格）**：會被當成一個「格位」占用，但無法選擇，產生
    無意義的空格
- **控制字元**：可能導致 UI 渲染錯亂
- **純 ASCII 字元（除非特別需要）**：九格輸入法主要用於漢字，混入 ASCII 會讓候選字
    不自然

**可以出現的字元**：

- 所有 CJK 字元（BMP 區 4E00-9FFF、擴展 A 3400-4DBF、擴展 B~F 在補助字面）
- CJK 符號（3000-303F、FF00-FFEF）
- HKSCS PUA 字元（E000-F8FF）
- `*`（佔位符，不會顯示）

## 五、載入行為與實作細節

### 5-1. 載入時機

程式啟動 → `App.xaml.cs` 的 `OnStartup` → 觸發 `CharTables.LoadFromResources` →
**一次性** 把整份 `char_map.txt` 讀進一個 `Dictionary<string, string>`，然後
**凍結**成 `FrozenDictionary`（.NET 的高效能只讀雜湊表）供後續使用。

所以：

- **檔案在程式啟動後即使被修改，對程式運作無影響**
- **重新啟動程式**才會套用新字庫

### 5-2. 效能：Prefix2Cache

啟動時，程式會額外建立一張 `Prefix2Cache`：掃描所有 3 碼 key，把它們按「前 2 碼」聚合。
例如：

```
110,華萬夢若著
111,著蓋萊藝苦
112,艾勸惹若英
113,繭藤藍蘭蕭
```

會聚合成：

```
"11" → ["華", "萬", "夢", "若", "著", "蓋", "萊", "藝", "苦", "艾", "勸", "惹", "英",
        "繭", "藤", "藍", "蘭", "蕭", ...]
```

此時使用者若在 SecondDigit 狀態（已輸入 `1`, `1`）點「確定」，程式 O(1) 查 `"11"` 直接
拿到所有 `11X` 的候選字，不用掃整張 map。

**對 `char_map.txt` 作者的意義**：你**不需要**手動維護 2 碼 key 的 `xx,...` 行。2 碼查詢
自動從 3 碼聚合而來。

### 5-3. 處理 `*` 時的自動去除

`Prefix2Cache` 聚合時會**自動丟棄** `*` 字元。意思是：

```
110,**華**萬
111,**夢**
112,*著*
```

聚合到 `"11"` 底下會是 `["華", "萬", "夢", "著"]`——`*` 的佔位只在單獨 3 碼查詢時有意義，
聚合為 2 碼查詢時已無格位意義。

另外，聚合時也會**去重**：

```
110,華萬
111,華夢
```

`"11"` 的聚合結果是 `["華", "萬", "夢"]`，不會重複出現「華」。

### 5-4. 判斷 grapheme（字素）

程式讀字時用 `StringInfo.GetNextTextElement`，這會正確處理：

- **基本 BMP 字元**：1 個 C# char（2 bytes UTF-16）
- **輔助字面（Supplementary）字元**：如「𠀀」（U+20000，CJK 擴展 B），由 surrogate pair
    組成，占 2 個 C# char（4 bytes UTF-16），算作 1 個 grapheme
- **組合符號**：理論上也會被正確合併，但實務上 `char_map.txt` 裡的字都是獨立 NFC
    正規化的，應該不會遇到

所以你可以安心在字庫裡寫「𠀀」「𠁀」這類擴展字元，它們會佔 1 個格位。

## 六、與其他字庫的關係

### 6-1. 與 `rel_map.txt` 的關係

`char_map.txt` 決定**怎麼查到字**；`rel_map.txt` 決定**查到字後下一個字是什麼**。

這兩份表沒有強制同步要求，但**最好保持一致性**：如果 `char_map.txt` 裡某字的編碼方式
和 `rel_map.txt` 裡該字的關聯字群完全不配套，使用者可能會看到「關聯字預覽顯示某字，
但按 `0` 進入選字後找不到它」的困惑。

### 6-2. 與 `cj5_map.txt` / `b5_map.txt` 的關係

這兩張表只在使用者啟用「轉倉」「轉易」輸出模式時才會用到。查詢路徑：

```
char_map.txt → 使用者選中「華」
轉倉輸出時：查 cj5_map.txt[「華」] → 得到倉頡碼 → 送 VK 事件 → 系統倉頡 IME 組字
```

所以 `cj5_map.txt` 裡收錄的字應該**包含**你的 `char_map.txt` 裡的所有字。
如果某個字在 `char_map.txt` 有但 `cj5_map.txt` 沒有，使用轉倉輸出時該字會自動退回
剪貼簿模式（仍能輸出，但體驗降低）。

### 6-3. 與 `b5_uni.txt` 的關係

`char_map.txt` 裡如果有 HKSCS PUA 字（U+E000~U+F8FF），這些字在輸出時會自動用
`b5_uni.txt` 轉成標準 Unicode。所以你的 `b5_uni.txt` 應該包含你所有 PUA 字。

程式會以**紅色**渲染 PUA 字，提醒你「這個字會經過自動轉換」。

### 6-4. 與 `t2s.txt` 的關係

同樣，如果使用者啟用「輸出簡體」，選了「華」會查 `t2s.txt["華"]` = "华"，然後送出「华」。
`t2s.txt` 沒有對應的字保持原樣輸出（繁體）。

```
30,周馬崔關歐區岑滕尹閔駱山鳳聞閻...
40,梁馮廖方唐許施郭曾鄭高江顏庄謝凌沈...

80,水史常卜米婁虞盧裴卓黨申冉冀尚農...
90,錢鄭衛朱何劉魏徐黎曾邱倪包伍...
110,華萬夢若著花落英苦幕蘭舊...
111,著蓋萊藝苦葉菁藉菱葦蕙薔茉葫...
112,艾勸惹若英茂藏芙茹莽茄荐荻荔...
113,繭藤藍蘭蕭芽蒙茁葷藺苒芃苣菔芟芑
114,荒蓮薪落薦芳茫菩薄莎芒蓉萍蕩...

```

分析：

- **`0,...`（標點）**：9 個最常用標點，剛好一頁，無需分頁
- **`09,...`（擴充標點）**：龐大的擴充字集，包含全形 ASCII、希臘字母、製表符等
- **`10,...` ~ `90,...`（百家姓）**：每個首碼下有一個姓氏列表，DEMO 版包含較完整的
    《百家姓》
- **`110,...`（首碼 1 首 2 碼 1 末碼 0）**：1xx 的通用字，末碼不精確指定
- **`111,...` ~ `119,...`**：同樣首 2 碼下，依末碼細分
- **`123,**期*`**：使用 `*` 佔位，保留「期」在第 3 格

## 八、作者的實務建議

### 8-1. 從 DEMO 擴充 vs 從零開始

- **從 DEMO 擴充**：複製內建版本，逐步加字。缺點是繼承了原作的拆碼風格，若你想做出
    獨特的分派會掣肘
- **從零開始**：制定一份完整的拆碼規範，再按規範建表。工作量大但可控

### 8-2. 建議工作流

1. 先確定**第 1 碼的 9 種筆型**語意，並畫出 `0s.png`
2. 從最高頻 500 字開始，逐一拆碼
3. 依拆出的碼分組，填進 `char_map.txt`
4. 實測：跑起程式，打 20 個字試試看碼是否直覺
5. 根據測試結果調整，再擴展到 2000 字、5000 字

### 8-3. 使用試算表管理大表

建議用 Excel / Google Sheets 管理：

| 字 | 1 碼 | 2 碼 | 3 碼 | code | 頻度分 |
|---|---|---|---|---|---|
| 華 | 1 | 1 | 0 | 110 | 8734 |
| 萬 | 1 | 1 | 0 | 110 | 7621 |
| 夢 | 1 | 1 | 0 | 110 | 4392 |
| 著 | 1 | 1 | 0 | 110 | 12903 |
| 著 | 1 | 1 | 1 | 111 | 12903 |
| ... | | | | | |

然後用以下 Python 片段產生 `char_map.txt`：

```python
import csv
from collections import defaultdict, OrderedDict

rows = defaultdict(list)  # code → [(char, freq)]
with open("char_map.csv", encoding="utf-8") as f:
    for r in csv.DictReader(f):
        code = r["code"]
        char = r["字"]
        freq = int(r["頻度分"])
        rows[code].append((char, freq))

with open("char_map.txt", "w", encoding="utf-8") as out:
    # 排序 code：0, 09, 10..90, 110..999
    for code in sorted(rows.keys(), key=lambda x: (len(x), x)):
        chars = sorted(rows[code], key=lambda p: -p[1])  # 高頻在前
        chars_str = "".join(c for c, _ in chars)
        out.write(f"{code},{chars_str}\n")
```

### 8-4. 版本管理

建議用 Git 追蹤你的 `char_map.txt` 演進。每次新增一批字就 commit，這樣回溯很方便。
**不要**直接 commit 到本程式的 GitHub 專案（因為該專案不收受可能有版權爭議的字庫），
用自己的 private repo 或獨立專案。

## 九、常見錯誤診斷

| 症狀 | 可能原因 |
|---|---|
| 打某碼沒有候選字 | 該 code 不存在於表中，或字串是空的 |
| 打某碼後顯示亂碼 | 檔案編碼不是 UTF-8，用支援 UTF-8 的編輯器重存 |
| 候選字順序和預期不同 | 檢查 `<chars>` 裡的字元順序是否符合你想要的格位 |
| 程式啟動變慢 | 字庫過大（>5 MB）會影響啟動時間。考慮切分 |
| 第 3 格一直是空的 | `<chars>` 前 2 個字是 `**`，卻沒用 `*` 佔位，自然就空 |
| 某碼應該有 20 字但只顯示 9 個 | 多出來的字在第 2 頁，按 `+` 翻頁即可看到 |

## 十、快速參考卡

```
格式：<code>,<chars>
編碼：UTF-8
code：1~3 位 ASCII 數字，不可有前導 0（除了 code = "0" 本身）
chars：任意 CJK 字元串，順序 = 格位 (1→格 1 左下 … 9→格 9 右上)
特殊字元：*（佔位，不顯示、不可選）
重複 key：只留最後一筆
空行：允許，直接略過
```
